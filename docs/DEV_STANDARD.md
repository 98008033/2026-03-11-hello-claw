# 开发标准与工作流程

## 0. 开发审批流程（重要！）

**后续开发必须严格执行：**

```
1. 需求分析 → 输出 PRD.md
2. 制定方案 → 输出 PLAN.md  
3. Commit 到 GitHub
4. ⏸️ 等待用户批准
5. ✅ 批准后 → 开始开发
6. 开发完成 → 提交并关联 Issue
```

**⚠️ 禁止事项：**
- 未批准前不得开始开发
- 不得跳过方案评审直接 coding
- 紧急 BUG 修复需事后补 Issue

## 1. GitHub Issues 工作流程

所有问题、需求变更、BUG 修复、功能优化必须通过 GitHub Issues 跟踪。

### 1.1 创建 Issue

**触发条件:**
- 发现 BUG
- 用户需求变更
- 新功能请求
- 性能优化
- 代码重构
- 文档更新

**Issue 标题格式:**
```
[类型] 简短描述
```

**类型标签:**
- `bug` - BUG 修复
- `feature` - 新功能
- `enhancement` - 现有功能优化
- `docs` - 文档更新
- `refactor` - 代码重构
- `performance` - 性能优化

**Issue 模板:**
```markdown
## 问题描述
清晰描述问题或需求

## 复现步骤 (BUG 类)
1. 步骤 1
2. 步骤 2
3. 预期结果 vs 实际结果

## 期望结果
期望的修复效果或功能表现

## 影响范围
- 前端/后端/文档
- 具体文件/模块

## 优先级
- P0: 紧急阻塞
- P1: 高优先级
- P2: 中优先级
- P3: 低优先级
```

### 1.2 Issue 处理流程

```
创建 Issue → 分配标签 → 开发修复 → 提交关联 → 测试验证 → 关闭 Issue
```

**步骤说明:**

1. **创建 Issue**
   - 使用统一模板
   - 添加合适标签
   - 设置优先级

2. **分配标签**
   - `bug` / `feature` / `enhancement` 等类型标签
   - `frontend` / `backend` / `docs` 等模块标签
   - `P0` / `P1` / `P2` / `P3` 优先级标签

3. **开发修复**
   - 在 Issue 中评论修复方案
   - 创建分支（可选）：`fix/issue-#123` 或 `feature/issue-#123`
   - 进行代码修改

4. **提交关联**
   - Commit message 必须包含 Issue 引用
   - 格式：`fix: 描述 (close #123)` 或 `feat: 描述 (ref #123)`
   - 示例：
     ```
     fix: 按钮 mega 模式丢失立体效果 (close #45)
     - 恢复原始橙黄渐变
     - 添加内阴影保持立体感
     - 只增强外部发光
     ```

5. **测试验证**
   - 在 Issue 中评论测试结果
   - 附上测试截图或录屏（可选）
   - 确认问题已解决

6. **关闭 Issue**
   - 使用 commit 自动关闭：`close #123`
   - 或手动关闭并说明
   - 更新 PLAN.md 中的任务状态

### 1.3 示例流程

**示例 1: BUG 修复**
```
1. 创建 Issue #45: [bug] 50 连击按钮变红但丢失立体效果
2. 添加标签：bug, frontend, P1
3. 评论修复方案：恢复渐变 + 添加内阴影
4. 提交代码：fix: 恢复按钮 3D 效果 (close #45)
5. 评论：已验证，立体效果恢复
6. Issue 自动关闭
```

**示例 2: 功能优化**
```
1. 创建 Issue #32: [enhancement] 提升粒子特效范围
2. 添加标签：enhancement, frontend, P2
3. 评论方案：粒子距离 400→600px，数量×2
4. 提交代码：feat: 粒子特效全面增强 (close #32)
5. 评论：附测试截图，特效明显提升
6. Issue 自动关闭
```

## 2. 代码提交规范

### 2.1 Commit Message 格式

```
<type>: <description> (<issue-reference>)

<optional-body>
```

**Type 类型:**
- `fix` - BUG 修复
- `feat` - 新功能
- `enhancement` - 功能优化
- `docs` - 文档更新
- `refactor` - 代码重构
- `performance` - 性能优化
- `style` - 代码格式（不影响功能）
- `test` - 测试相关

**示例:**
```bash
# 修复 BUG
fix: 按钮 mega 模式丢失立体效果 (close #45)
- 恢复原始橙黄渐变
- 添加内阴影保持立体感

# 新功能
feat: 添加 100 连击闪电风暴特效 (close #38)
- 20 道主闪电
- 分支闪电效果
- 100 个闪电粒子

# 功能优化
enhancement: 提升粒子数量 (close #32)
- 基础粒子 60→180
- 粒子范围扩大 2 倍

# 文档更新
docs: 更新部署指南 (ref #50)
- 添加 Render 配置说明
- 补充环境变量配置
```

### 2.2 分支管理（可选）

简单项目可直接在 `main` 分支开发，复杂项目使用分支：

```bash
# 创建分支
git checkout -b fix/issue-45

# 开发完成后合并
git checkout main
git merge fix/issue-45
git push origin main
```

## 3. 文档更新规范

### 3.1 必须更新的文档

| 文档 | 更新时机 |
|------|----------|
| `PLAN.md` | 每个阶段完成/状态变更 |
| `CHANGELOG.md` | 每次发布/重大更新 |
| `README.md` | 项目信息变更 |

### 3.2 PLAN.md 更新

每次任务完成后更新状态：

```markdown
## 2. 任务分解

### 2.2 前端任务

| 任务 | 状态 | 工时 | 说明 |
|------|------|------|------|
| 粒子特效优化 | ✅ 完成 | 1h | Issue #32 |
```

### 3.3 CHANGELOG.md 格式

```markdown
# 更新日志

## [0.2.0] - 2026-03-12

### Added
- 100 连击闪电风暴特效 (#38)
- 分支闪电效果 (#38)

### Changed
- 粒子数量提升 2 倍 (#32)
- 粒子范围 400→600px (#32)

### Fixed
- 按钮 mega 模式丢失立体效果 (#45)
- Combo 框过度放大问题 (#46)
```

## 4. 测试验证

### 4.1 自测清单

提交前确认：
- [ ] 代码可正常运行
- [ ] 修复的问题已验证
- [ ] 无新增 BUG
- [ ] 相关文档已更新

### 4.2 测试记录

在 Issue 中评论测试结果：

```markdown
## 测试验证

- [x] 本地测试通过
- [x] 生产环境验证
- [ ] 回归测试

**测试截图:**
（可选）
```
